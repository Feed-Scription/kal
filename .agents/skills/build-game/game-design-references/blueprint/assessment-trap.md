# 测评表陷阱（Critical Anti-pattern）

**症状**：游戏玩起来像心理测评——玩家反复做同一种形式的选择题，每次都是"读一段描述 → 从几个选项中选一个 → 看数值变化 → 下一题"。

**根因**：核心循环只有一种交互形式（选择题），且选择的后果仅体现为数值变化，不影响后续可用的场景、NPC 行为或叙事走向。

**检验方法**：把核心循环写成伪代码。如果整个游戏可以简化为：

```
repeat N times:
  show_text(event_description)
  choice = pick_option(options)
  update_stats(choice.effects)
```

那就是测评表，不是游戏。

## 通过标准

质量检查清单 #3 和 #4 的具体要求：

核心循环必须包含至少 2 种以下交互形式：
- 选择题式决策（从选项中选一个）
- NPC 对话（多轮、有记忆、影响关系）
- 自由行动（玩家主动选择做什么，而非被动响应事件）
- 资源分配（分配时间/金钱/精力到不同方向）
- 探索/调查（主动获取信息，解锁新内容）
- 场景导航（在不同地点/场景间移动）

同时，选择的后果必须至少通过以下一种方式体现持久影响：
- 解锁或封锁后续场景/事件
- 改变 NPC 的态度或可用对话
- 触发因果链（A 选择导致 B 事件在 N 回合后发生）
- 改变可用的行动选项

## 反面案例

设计文档写了"NPC 关系网络"和"对话系统"，但核心循环只写了"月度事件 → 选选项 → 属性变化"。NPC 和对话没有融入循环，实现时自然被忽略，最终产出的游戏就是 36 道选择题。

## 正面做法

核心循环应写成"进入场景 → 选择行动（探索/对话/事件）→ 行动产生后果（关系变化/解锁新场景/触发事件链）→ 推进到下一时段"。NPC、对话等元素必须作为循环的一个步骤出现，而非只在"游戏元素"表格里列出。
