# KAL-AI 市场调研与竞品分析报告

> 调研时间：2026 年 2 月 28 日

## 一、核心结论

**KAL-AI 要解决的问题是真实存在的，市场时机正确，但窗口期有限。**

1. AI 原生游戏正在从小众实验变成一个真实品类，中国市场尤为活跃
2. 开发者反复提及的痛点（多模型调度、延迟、成本、AI 幻觉、缺乏测试手段）与 KAL-AI 的设计高度对应
3. 在"面向程序员的、TypeScript 的、专为 AI 原生游戏设计的开源编排框架"这个定位上，目前没有直接竞品
4. 竞争来自两个方向：上层的无代码游戏平台（SOON/Gambo/Rosebud），和下层的 AI 文游平台（MOKU/仙途/XStory）
5. 最大的风险不是市场不存在，而是执行速度——其他方案可能更快占位

## 二、AI 原生游戏市场全景

### 2.1 什么是 AI 原生游戏

AI 原生游戏是指以 LLM/生成式 AI 为核心游戏机制（而非辅助功能）的游戏。去掉 AI，游戏不存在。

与之对比：
- **AI 辅助游戏**：传统游戏用 AI 生成美术资产、对话、关卡（AI 是工具）
- **AI NPC 游戏**：传统游戏加入 AI 驱动的 NPC 对话（AI 是功能模块）
- **AI 原生游戏**：游戏的核心循环就是 AI 生成（AI 是产品本身）

## 三、开发者真实痛点

以下痛点全部来自开发者公开采访和技术文章，非推测。

### 3.1 多模型混合调度没有标准工具

> "我们先设计了几个后端请求API接口，每个接口对应不同模型或同一模型，再根据性能和速度综合考量进行组合。" ——青椒模拟器开发者专访

每个 AI 游戏开发者都在手动搭建多模型调度逻辑。高成本模型负责核心内容生成，低成本模型负责边缘内容（如朋友圈、NPC 介绍），但这套调度完全靠手工胶水代码。

**KAL-AI 对应方案**：`ModelFactory`（多模型注册与管理）+ Flow 多 LLM 节点（声明式编排）

### 3.2 响应延迟严重影响体验

> "用AI做游戏的最大难点就是响应慢，玩家输入后可能要等十几到三十秒才能得到结果。" ——青椒模拟器开发者专访

10-30 秒的等待是 AI 游戏体验的头号杀手。Intra 的开发者也在 wishlist 里写到希望能"precalculate while waiting for input"和"find parallelism"。

**KAL-AI 对应方案**：三级优先级（`interactive` / `prefetch` / `background`）、并行调度（`Scheduler`）、流式输出

### 3.3 Token 成本失控

> 青椒模拟器：自掏数万元购买 Token，后获 Kimi 赞助
> Death by AI：首月消耗 12 亿 token，使用 OpenAI + ElevenLabs 无法盈利
> 独立开发者普遍反馈："太贵了，搞不起"

成本是当前 AI 原生游戏最大的瓶颈。好消息是 LLM 推理成本正以每年 10 倍的速度下降（GPT-4 级别：2022 年 $20/M tokens → 2026 年 $0.40/M tokens），但缓存和成本优化仍然是刚需。

**KAL-AI 对应方案**：三层缓存（L1 内存 / L2 本地 / L3 远程）、`poolPolicy`、`CacheKeyStrategy`

### 3.4 AI 幻觉和内容质量不可控

> 《崇祯》中出现在 1627 年提及 1644 年才建立的"大顺"的错误
> AI 文游评测："字里行间的套路感实在太重了"、"AI 记忆池比较小"

AI 幻觉在游戏中尤其致命——它直接破坏世界观一致性。

**KAL-AI 对应方案**：`StateManager`（约束世界状态）、`state-read` 节点（向 prompt 注入结构化上下文）、`SafetyFilter`、`JsonRepair`

### 3.5 数据收集和可观测性缺失

> "前端请求的数据种类多、类型不同，还要埋点收集大模型的请求返回结果。又不想用关系型数据库，最后只能用一些不太完善的方法暂存数据。" ——青椒模拟器

**KAL-AI 对应方案**：`Observer` + `UsageTracker`

### 3.6 没有标准化测试手段

目前所有 AI 原生游戏案例中，没有任何一个提到过系统性测试。全部是"上线看看"。

**KAL-AI 对应方案**：DevKit（Simulator 模拟玩家、A/B Prompt/Model 测试、Recorder/Replayer、Inspector 断言）

### 4.2 AI 文游平台（最接近的生态位）

#### 小兔文游世界（xiaotuyouxi.cn）

- **定位**：AI 文游聚合平台 + 虚拟货币变现，Web 端
- **功能**：平台上架 12+ 款 AI 文游（小三上位模拟器、年代文婚后模拟器、换乘恋综模拟器、离婚综艺模拟器等），另有"自由剧本"模式（玩家自定义任意场景）
- **目标用户**：女性向为主，几乎全部是恋爱/情感/宫斗/综艺题材
- **商业模式**：
  - "萝卜"虚拟货币：1 根萝卜 = 1 次 AI 对话（即 1 次 LLM 调用）
  - 全站通行证 ¥19.9（解锁所有游戏），畅玩包 ¥22-33（通行证 + 萝卜）
  - 萝卜续费：150 根 ¥3、520 根 ¥9、1000 根 ¥15
  - 也支持用户自带 API Key（免费使用，面向技术用户）
- **开发者**：个人开发者（"小兔第一次做游戏+卖萝卜，啥都在摸索中"），通过小红书和 QQ 群运营
- **技术细节**：Web 前端，后端调用 LLM API（支持 OpenAI 兼容端点 /v1），有 JSON 解析失败重试机制
- **与 KAL-AI 的关系**：
  - 小兔文游是一个**已跑通商业模式的 AI 文游产品**——虚拟货币包装 token 成本，让玩家无感付费
  - 其技术栈（Web + LLM API + JSON 解析）正是 KAL-AI 覆盖的范围
  - 痛点直接可见：JSON 解析失败（KAL-AI 的 `JsonRepair`）、API 调度（`ModelFactory`）、成本控制（缓存）
  - 代表了一类**从 Prompt 售卖升级到平台化运营**的 AI 文游创作者，是 KAL-AI 的精确目标用户

#### MOKU 墨盒（mokulab.cn）

- **定位**：高自由文游模拟器，Web 平台
- **功能**：用户可以创建和游玩 AI 文字游戏，有分类标签（修仙仙侠、古风、奇幻穿越、末日生存、校园、都市等）
- **数据**：热门剧本有 20 万+ 游玩次数（如"高中生活模拟器" 202.2k、"大学校花模拟器" 141.5k）
- **商业模式**：平台制，UGC 模式，创作者在平台上发布剧本，有 QQ 交流群（1078599949）
- **技术**：服务器在海外，提供加速选项
- **与 KAL-AI 的关系**：MOKU 是 AI 文游的**消费/UGC 平台**（类似 Steam 之于游戏），KAL-AI 是**开发框架**（类似 Unity 之于游戏开发）。不是竞品，而是潜在的生态伙伴——MOKU 上的创作者可能是 KAL-AI 的用户

#### 仙途（ddct.top）

- **定位**：AI 驱动的修仙文字冒险游戏
- **功能**：支持 Gemini、Claude、OpenAI、DeepSeek 多模型；NPC 独立思维链；开放式社区创作；完整修仙体系
- **版本**：V4.5（2026.01），已有宗门战争模拟系统
- **与 KAL-AI 的关系**：仙途是一个**特定品类的 AI 游戏产品**，不是框架。但它的多模型支持、NPC 思维链、世界预设系统等技术需求，正是 KAL-AI 要解决的

#### XStory（xstory.io）

- **定位**：AI 驱动的写作工作室 / 互动叙事平台
- **功能**：创作者可定义物理规则、魔法系统和背景故事；支持分支叙事和自动调整；面向创作者和玩家的双边平台
- **与 KAL-AI 的关系**：XStory 是更偏向"互动小说"的平台，但其底层的世界规则定义、分支叙事生成等能力，在技术上可以由 KAL-AI 支撑

#### gapp.so

- **定位**：AI 生成游戏的展示画廊 / 发行平台
- **Slogan**："Games and apps, built by AI. No download, no setup — just play."
- **功能**：浏览和游玩 AI 生成的浏览器游戏，支持开发者提交作品
- **与 KAL-AI 的关系**：纯分发平台，不涉及游戏开发技术。KAL-AI 构建的游戏可以在 gapp.so 上发布

#### AI 文游平台对比总结

| 维度 | **小兔文游世界** | **MOKU 墨盒** | **仙途** | **XStory** | **gapp.so** |
|------|-----------------|---------------|----------|------------|-------------|
| **模式** | 聚合平台 + 虚拟货币 | UGC 平台 | 单品游戏 | 创作平台 | 画廊/分发 |
| **内容来源** | 平台主自制 | 用户创作 | 开发者自制 + 社区预设 | 创作者 + 玩家 | 开发者提交 |
| **变现** | 通行证 ¥19.9 + 萝卜（¥0.015/次对话） | 不详 | 用户自带 API | 不详 | 免费 |
| **技术栈** | Web + LLM API | Web | 多模型（Gemini/Claude/OpenAI/DeepSeek） | Web | Web |
| **品类** | 女性向恋爱/综艺 | 修仙/古风/校园等 | 修仙 | 泛叙事 | 泛游戏 |
| **创作者门槛** | 低（平台主运营） | 中（写剧本） | 中（写世界预设） | 中（定义规则） | 高（需开发） |
| **与 KAL-AI 关系** | 目标用户 | 生态伙伴 | 技术需求匹配 | 底层可支撑 | 分发渠道 |

**关键洞察**：这些平台共同勾勒出一个正在成型的产业链——

```
Prompt 创作者 → 平台化运营（小兔/MOKU） → 更复杂的 AI 游戏产品（仙途/崇祯）
                          ↑
                   KAL-AI 的切入点
                （当简单 Prompt 不够用，需要多步编排、状态管理、缓存优化时）
```

小兔文游世界是一个特别值得关注的案例——它已经走到了"Prompt → 产品 → 平台"的第三步，正在用虚拟货币包装 token 成本，而且明确遇到了 JSON 解析失败等技术问题。当这类平台想要升级游戏质量（更复杂的世界状态、多 NPC 交互、数值系统），KAL-AI 就是自然的下一步。

#### AI 文游 Prompt 生态（小红书）

- **定位**：去中心化的 AI 文游创作和交易生态
- **形态**：创作者在小红书售卖 Prompt PDF，价格 2.99-29.9 元
- **规模**：单链接最高销量近 5000；有专职博主和社区
- **演进趋势**：从 PDF → 网页版 → 平台化
- **与 KAL-AI 的关系**：这些创作者正在从"写 Prompt"升级到"做网页游戏"。当他们需要超越纯 Prompt 的能力（多步管线、状态管理、数值系统），KAL-AI 就是他们的下一步工具

### 4.3 无代码游戏生成平台（间接竞品）

| 平台 | 类型 | 引擎 | 特点 | 与 KAL-AI 的区别 |
|------|------|------|------|-------------------|
| **SOON/极逸** | 全要素 AI 游戏制作 | Unity + Three.js | 2D/3D，14 天可出微信小游戏 | 面向非程序员，生成标准小游戏 |
| **Gambo** | 无代码游戏生成 | Web | 文本描述 → 可玩 2D 游戏 | 模板化，非 AI 原生 |
| **Rosebud** | 浏览器游戏生成 | Three.js/Phaser | 教育和原型 | 简单，非生产级 |
| **Mirage** | World Model 游戏引擎 | 自研 | 实时世界模型驱动 UGC | 不同范式（视觉生成 vs LLM 编排） |

**关键区别**：这些平台用 AI 来**生成游戏**（AI 是制作工具），而 KAL-AI 用 AI 来**驱动游戏**（AI 是运行时核心）。两者服务的是完全不同的需求。

### 4.4 通用 AI 框架（技术层竞品）

| 框架 | 语言 | 定位 | 游戏适配性 |
|------|------|------|------------|
| **LangGraph** | Python (+ JS) | 通用 AI Agent 编排 | 低：循环图模型适合 Agent 不适合游戏管线；Python 生态 |
| **Vercel AI SDK** | TypeScript | AI 应用开发工具包 | 中：TypeScript 原生，但无游戏概念 |
| **OpenAI Agents SDK** | Python/TS | AI Agent 框架 | 低：面向 Agent，无游戏状态/交互模型 |
| **Inworld Runtime** | C++ / Node.js | 实时 AI 基础设施 | 中：有图编排，但已转向通用平台 |

**KAL-AI 的独特位置**：目前不存在一个同时满足以下条件的开源框架：
1. TypeScript 原生
2. 浏览器 + Node.js 双端
3. 游戏专用（状态管理、交互节点、优先级调度）
4. DAG 编排 + 多模型支持
5. 配套测试工具链（模拟玩家、A/B 测试、录制回放）

| 特征 | 证据 |

- **AI 文游创作者**：正在从 Prompt 升级到网页版的创作者

### 6.2 KAL-AI 自身的潜在商业模式

作为开源框架，KAL-AI 本身不直接变现。但有几条可能路径：
- **增值云服务**：托管的缓存、模型路由、可观测性
- **企业版 DevKit**：面向游戏公司的测试和 CI/CD 工具
- **平台合作**：为微信小游戏/抖音等平台提供 AI 游戏开发工具链

### 6.3 Token 成本趋势

LLM 推理成本正在指数级下降，这是 AI 原生游戏最大的利好：

| 时间 | GPT-4 级别性能成本（/M tokens） | 降幅 |
|------|-------------------------------|------|
| 2022 年底 | $20.00 | - |
| 2024 年中 | $4.00 | 5x |
| 2026 年初 | $0.40 | 50x |
| 2026 年底（预测） | ~$0.07 | ~280x |

## 七、技术趋势与风险

### 7.1 有利趋势

**成本崩塌**：每年 10 倍的成本下降使更复杂的 AI 管线变得可行。当单次 LLM 调用近乎免费时，开发者会从"省着用"转变为"狠狠用"，复杂编排的需求自然涌现。

**端侧小模型成熟**：Gemma 3 270M、Llama 3.2 1B、Qwen 3 4B 等可在消费级设备上运行的模型，支持 function calling。端侧 + 云端混合编排是 KAL-AI 的潜在使用场景。

**微信生态主动推动**：微信 2026 年推出"AI 小程序成长计划"，免费提供 1 亿 Token + 云开发资源。平台方在培育 AI 原生小游戏供给。

**Vibe Coding 降低门槛**：非游戏开发者用自然语言生成代码的能力爆发，产生大量"有想法但缺基础设施"的开发者——正是 KAL-AI 的目标用户。

### 7.2 风险

**World Model 颠覆**：Google Genie 3 已经能实时生成可交互 3D 世界（720p/24fps）。如果 World Model 在 2-3 年内成熟，"用 LLM 生成文本叙事"的范式可能被"直接生成可玩世界"取代。但短期内（2026-2027），Genie 3 限制仍大（60 秒一致性、多角色交互差），不构成直接威胁。

**平台方自建**：微信/抖音可能自建 AI 游戏开发工具链，直接覆盖 KAL-AI 的功能。但平台方通常做通用工具而非垂直框架，且开源框架的灵活性是平台方案难以替代的。

**市场教育成本**：AI 原生游戏的开发者多为个人或小团队，他们可能更倾向于用最熟悉的工具（直接调 API）而非学习新框架。KAL-AI 需要证明"用框架比手写快"。

**品类天花板不确定**：85% 玩家对 AI 游戏持负面态度（担心低质量 "gameslop"）。AI 原生游戏能否成为主流品类，还是始终是小众实验，仍有不确定性。

## 八、FeedScription——团队自建的分发平台

### 8.1 产品发现

FeedScription（platform.feedscription.com）是 KAL-AI 团队自建的 AI 游戏分发平台，Slogan 为 **"Connecting AI App Creators with Users — Making AI Apps Accessible"**。

目前上架产品：

| 产品 | 作者 | 价格 | 状态 |
|------|------|------|------|
| 青椒模拟器 V1.5 | 青椒模拟器制作组 | 200 Credits（-90%） | 在售 |
| 青椒模拟器 V1.5 离线版 | 青椒模拟器制作组 | 免费 | 可玩 |
| Presiduck Simulator | pupu | 249 Credits（-90%） | 在售 |
| Fate River | hoho | 150 Credits（-90%） | 在售 |

其中青椒模拟器是朱桐和周厚全的作品（累计 60 万玩家），Fate River 在 KAL-AI 仓库中有对应的技能文件（"Add a new playable character to fate-river"），确认为团队自研产品。

### 8.2 商业模式分析

FeedScription 采用 **Credits 虚拟货币**制：
- 用户购买 Credits → 用 Credits 购买/游玩 AI 游戏
- 类似小兔文游的"萝卜"，但定位为**开放平台**（不只卖自己的游戏）
- 已提供离线版选项（降低持续 token 成本）

### 8.3 战略意义

团队构建了一个**完整的垂直栈**：

```
分发层：FeedScription（AI 游戏商城 + Credits 变现）
    ↑ 游戏在此发布和销售
产品层：青椒模拟器 / Fate River（自研 AI 原生游戏）
    ↑ 使用框架构建
框架层：KAL-AI（@kal-ai/core + @kal-ai/devkit）
    ↑ 编辑和调试
工具层：KAL-AI UI（Agent-first IDE）
```

这个策略的优势：
1. **自己吃自己的狗粮**：团队用 KAL-AI 构建自己的游戏，确保框架解决真实问题
2. **双飞轮效应**：框架降低开发门槛 → 更多游戏上架 FeedScription → 更多用户 → 更多开发者使用框架
3. **不依赖单一收入**：框架开源吸引开发者，平台通过 Credits 抽成变现
4. **与小兔文游/MOKU 等竞品的差异**：小兔文游是封闭平台（只卖自己的游戏），FeedScription 定位为开放市场

这也解释了为什么 KAL-AI 的设计文档如此详尽且接地气——设计者自己就是最深度的用户。

## 九、战略建议

### 9.1 优先级排序

1. **最快速度推出可用的 @kal-ai/core**（含 Model、State、Flow、Prompt）——让开发者能用起来
2. **做一个杀手级示例**——用 KAL-AI 构建一个类似青椒模拟器/崇祯的 AI 原生游戏，证明"用框架比手写快 10 倍"
3. **DevKit 可以延后**——测试工具有价值，但不是开发者的第一需求
4. **UI/编辑器是长期项目**——当前阶段不是优先级

### 9.2 定位建议

建议的 Slogan 方向：

> **KAL-AI：让 AI 原生游戏开发快 10 倍的 TypeScript 框架**

不要说"游戏 AI 框架"（容易被理解为给传统游戏加 AI NPC），而是说"AI 原生游戏开发框架"（明确是为以 AI 为核心的游戏设计的）。

### 9.3 生态策略

- 与 AI 文游平台（MOKU、仙途等）合作，提供技术支撑
- 在 AI 文游创作者社区（小红书）推广，从 Prompt 创作者转化为框架用户
- 与微信小游戏生态对接，利用平台的 Token 补贴和流量扶持

---

### 9.4 FeedScription 平台策略建议

鉴于团队已经有了分发平台，建议调整优先级：

1. **先让 FeedScription 跑起来**——用现有手工代码支撑，快速上架更多游戏，验证 Credits 商业模式
2. **从 FeedScription 的运营痛点反推 KAL-AI 的优先级**——哪些代码在多个游戏间重复？哪些 bug 反复出现？这些就是框架最先要抽象的部分
3. **邀请外部开发者入驻 FeedScription**——"用 KAL-AI 做游戏，在 FeedScription 卖"形成闭环
4. **参考小兔文游的定价策略**——小兔的 ¥0.02/次对话 定价被市场接受，FeedScription 的 Credits 体系可以参考
