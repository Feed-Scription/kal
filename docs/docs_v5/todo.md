# KAL 后续开发 Todo

**来源**：`文字记录：Cal游戏引擎开发会议总结 2026年3月11日.docx`

**整理时间**：2026-03-11

**目的**：把会议中已经讨论出的方向、共识、行动项和待确认问题整理成可执行的开发清单，供后续接力开发使用。

相关设计说明：

- [first-principles-architecture.md](./first-principles-architecture.md)

## 当前共识

1. KAL 当前已经不只是“AI 组件库”，而是在朝一个完整的 AI 原生游戏引擎演进。
2. 架构主线仍然成立：
   - `core` 负责运行时能力
   - `engine` 负责项目加载、CLI、HTTP API、TUI
   - `editor` 负责可视化查看与轻量编辑
3. 近期重点仍然是文字游戏 / 文游 / 回合制交互原型，不把主要精力放在重 UI 或复杂渲染上。
4. 前端 UI 不作为近期主战场，但需要至少做一个简单 Web 示例，把“Engine 如何和前端页面绑定”这条路走通。
5. `editor` 的定位需要上调，长期更接近 `studio`，但短期是否彻底并入、做到多大，还需要边做边验证。

## P0：当前最优先事项

### 1. 开源与文档整理

- 完成当前阶段的 README / 架构说明 / Todo 文档整理
- 把当前仓库整理到适合公开协作的状态
- 明确 CLI 命名和对外术语
  - 倾向统一为 `kal serve`、`kal run`、`kal studio`、`kal init`、`kal new`
- 补一份“项目文件怎么组织”的最小文档
- 补一份“如何启动 engine + editor/studio + example”的最小上手文档

### 2. 清理当前可见问题

- 修掉当前项目里已知的小 bug
- 对于较大、较重的 bug，先记录到 Todo，不阻塞开源
- 优先处理 editor 中已经观察到的显示问题
  - 节点信息显示不全
  - 节点状态/配置展示不够直观
  - 节点原始类型信息可见性不足

### 3. 验证 Engine 对外接前端的链路

- 实测 `kal serve` 提供的 API 是否足够支撑外部前端调用
- 梳理哪些接口已经够用，哪些接口还缺
- 明确外部前端与 engine 的推荐集成方式
  - 输入如何传给 flow / session
  - 输出如何回传给页面
  - state 如何读取或订阅
  - SignalIn / SignalOut 如何作为 UI 绑定契约暴露

### 4. 增加简单 Web 示例

- 新增一个最小化 Web 前端示例
- 示例目标不是做完整产品，而是验证“网页 UI + engine API + flow/session”的绑定方式
- 示例尽量简单
  - 选择题
  - 状态更新
  - 页面切换
  - 与后端 flow 执行联动

### 5. 扩充 examples

- 把示例数量扩展到 3-5 个
- 先做几个最小示例，再做一个体量稍大的示例
- 不同示例应覆盖不同用法，而不是同一种 DND 变体
- 目标是用 examples 反向发现框架缺口
  - 缺哪些节点
  - 缺哪些 API
  - TUI 哪些交互不够
  - Session 设计哪里不顺

## P1：下一阶段核心开发

### 1. DND 示例继续扩展

- 扩写当前 DND 示例的世界观、背景和主线内容
- 提升“把玩家拉回主线”的能力，避免剧情持续漂移
- 继续优化角色创建、背景补全、主循环叙事的体验
- 通过更复杂剧情验证 session + flow 的规模化可行性

### 2. TUI 能力增强

- 继续把 `kal play` 作为低成本试玩和验证入口
- 优化当前 TUI 对复杂玩法的支持
  - 选择
  - 状态查看
  - 帮助命令
  - 战斗/掷骰子/技能使用等分支交互
- 评估是否按既有计划推进到 Ink 版本
- 保持“先可玩，再美化”的原则

### 3. Studio 方向探索

- 评估是否将 `editor` 明确升级为 `studio`
- 短期至少先统一命名，减少 `editor / engine / studio` 的语义混乱
- 中期探索把运行态能力并入可视化工具
  - 在画布中看到当前执行到哪个节点
  - 节点高亮
  - Flow / Node 断点
  - 单步执行
  - 运行面板或停靠式调试面板
- 保留“单独运行 `kal run` / `kal play`”的轻量入口，不强依赖 studio

### 4. Engine 热更新与运行时联动

- 支持项目文件变化后的自动 reload
- 评估 `play` 模式下的热更新能力
- 为 studio 调试能力提供运行时事件输出
  - Flow 开始/结束
  - Node 开始/结束
  - 输入/输出
  - state 变化
- 可能需要新增事件流或调试专用接口

### 5. Flow / Session 进一步扩展

- 评估是否支持多 session
- 明确 session 的职责边界
  - 是否只负责游戏主循环编排
  - 是否允许拆成 UI / 存档 / 多模态等多个 session
- 评估动态生成 flow 的需求
  - 由 LLM 生成新的 flow
  - 运行时重新加载新 flow
  - 是否需要专门的“操作 flow”节点

## P2：生态与平台能力

### 1. 插件与分享机制

- 明确自定义 node 的共享方式
- 明确 flow / session 模板的共享方式
- 设计依赖打包关系
  - Node 是底层依赖
  - Flow 可能依赖 Node
  - Session 可能依赖 Flow 和 Node
- 形成一套可分发、可复用、可组合的模板/插件模型

### 2. Marketplace

- 在后续官网中预留 marketplace
- 支持上传和分享：
  - 自定义 Node
  - Flow 模板
  - Session 模板
  - 完整项目模板
- 需要先把格式、依赖关系和元数据规范清楚

### 3. AI 辅助开发能力

- 补一套让 AI 能直接生成 KAL 项目文件的文档或 skill
- 明确 AI 写项目时应该参考什么规范
  - `kal_config.json`
  - `flow/*.json`
  - `session.json`
  - `node/*.ts`
- 在框架相对稳定后整理成：
  - README
  - 示例模板
  - skill / SDK / prompt 规范

### 4. 发布与包管理

- 规划 npm 包名 / scope
- 待框架稳定后发布
- 发布前先清掉明显 bug 和不稳定接口

## 可能新增的能力点

- 掷骰子 / 概率判断节点
- 战斗系统相关节点
- UI 绑定相关底层节点或协议
- 多模态生成相关 flow / node
- 存档 / 读档能力
- 更强的调试与回放能力

## 暂时不建议重投入的方向

- 自己维护一整套通用前端 UI 组件库
- 提前投入复杂图形渲染或重视觉前端
- 在框架尚未稳定时过早固化完整 SDK 规范

## 建议的执行顺序

1. 整理文档、命名、开源前说明
2. 修小 bug，记录大 bug
3. 验证 engine API 与外部前端联动
4. 增加一个最小 Web 示例
5. 扩充 examples，覆盖更多玩法
6. 扩写 DND 示例并反向补框架能力
7. 推进 studio 调试能力和 engine 事件流
8. 再考虑 marketplace、插件生态和发布

## 待确认问题

- `editor` 是否直接改名并并入 `studio`，还是先保留现状、逐步增强
- `kal play`、`kal run`、`kal studio` 三者的最终职责边界
- 是否正式支持多 session
- 动态生成 flow 的能力是否进入近期范围
- Marketplace 的交付形式是官网能力、仓库模板，还是 npm 包生态

## 备注

- 近期应继续强调 KAL 的核心价值是“低成本验证 AI 游戏玩法与循环”，而不是做重型前端引擎。
- examples 和试玩体验会直接决定后续框架演进方向，优先级应高于过早做平台化包装。
