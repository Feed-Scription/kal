# 设计与文档演进：V1 → V5

## 演进脉络

### V1：全功能 SDK

V1 的重心是“能力全覆盖”。它更像一个通用 AI 基础设施 SDK，而不是聚焦游戏工作流的引擎。优点是想得很全，缺点是太重，门槛高。

### V2：节点式工作流引擎

V2 把编程式 API 转成了 Node + Flow 的声明式模型，用 JSON 描述工作流，使用门槛大幅下降。但基础设施被削得过薄，遇到 LLM 调用失败时缺少重试、缓存和可观测性。

### V3：节点式引擎 + 透明基础设施

V3 在 V2 的基础上补回真正必要的基础设施：重试、缓存、JSON 修复、Hook、配置分层。这个版本开始接近“可用运行时”。

### V4：模块化 + Web UI 契约导向

V4 做的不是一次大规模实现，而是一次**架构和文档前置**：

- 模块边界更清晰
- JSON-first 契约更明确
- 面向 Web UI 的 NodeManifest / schemaVersion / SubFlow 契约被提出来
- Engine、Editor、Core 三层关系被完整描述

问题也在这里：V4 的文档比代码走得更远，很多内容属于目标架构，而不是当前现实。

### V5：实现状态对齐

V5 的核心变化不是“新增了一大批功能”，而是：

**把文档重新拉回代码现实。**

V5 的目标是：

- 哪些已经做完，就明确写成已完成
- 哪些只做了一半，就明确写成部分完成
- 哪些还不存在，就明确写成未完成

## V4 到 V5 的关键变化

### 1. 文档角色变化

V4 更像目标架构文档。  
V5 是实现状态文档。

这意味着 V5 不再试图回答“理想系统应该长什么样”，而是优先回答：

- 当前仓库里到底有什么
- 哪些模块是真实可用的
- 哪些部分仍然只是设计

### 2. 模块关系从“目标链路”切换为“双视图”

V4 默认强调：

```text
Editor -> Engine -> Core
```

V5 同时保留两种视图：

目标架构：

```text
Editor -> Engine -> Core
```

当前实现：

```text
Editor -> 本地文件
Core -> 独立运行时
Engine -> 未实现
```

### 3. Engine 文档降级为现状说明

V4 中 Engine 的 CLI/API 设计非常详细。  
V5 不再把这些设计当成已存在功能，而是把 Engine 写成：

- 当前职责定义
- 当前缺失项
- 下一阶段最小落地范围

### 4. Editor 文档改为本地模式

V4 默认把 Editor 写成依赖 Engine 的 Web UI。  
V5 改为以当前实现为准：

- 纯前端
- 本地文件访问
- 可编辑，但不可运行和调试

### 5. Core 成为当前事实中心

在 V5 中，Core 不再只是整体架构的一部分，而是当前仓库中最可靠、最完整的事实来源。  
如果读者只想知道“仓库现在真正实现了什么”，Core 是第一入口。

## V5 的意义

V5 的价值不在于把愿景写得更大，而在于消除三种混淆：

1. **设计与实现的混淆**
2. **目标架构与当前状态的混淆**
3. **Editor 本地模式与服务模式的混淆**

这会直接降低以下成本：

- 新人理解仓库的时间
- 继续实现 Engine 时的认知偏差
- 根据过期文档误判功能成熟度的风险

## 当前版本应如何理解

如果把整个仓库压缩成一句话：

> 当前 KAL-AI 是一个已具备核心运行时能力的 Flow 引擎，加上一个本地可视化编辑器雏形；而完整的服务化宿主层仍待实现。

这就是 V5 试图建立的统一认知。

## V5 之后：已识别的改进方向

V5 完成了"文档对齐代码"的工作，但在对齐过程中也暴露了 Core 运行时的若干设计缺陷。这些缺陷不影响"能跑"，但影响"好用"。

核心矛盾是：很多本该是引擎内置能力的东西（State 读取、对话历史管理、prompt 组装），都需要用户在画布上手动搭节点和连线，导致 Flow 图很快变得复杂。

已识别的五个改进方向：

1. Prompt 与 State 声明式绑定，减少手动连线
2. Prompt 拼装感知消息角色，减少中间节点
3. Flow 支持多 Input / 多 Output，让 SubFlow 真正可组合
4. 对话历史作为内置状态自动管理，支撑多轮对话
5. Flow JSON 拆分为 meta 和 data 两层，关注点分离

这些改进的共同方向是：从"手动连线驱动"进化为"声明式 + 状态驱动"。
